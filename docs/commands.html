The terminal half of TabBrew

TabBrew is a Chrome extension for people running 100–200 tabs. This CLI is how a terminal — and the AI agent working in it — joins that conversation: it reads your exported tabs and puts a proposed change in front of you, in the browser, where you accept or deny it.

The one rule worth memorising: no command here can close, pin, group, or move a tab. The CLI only ever writes files and talks to a loopback port. Tabs move in Chrome, after you press Accept.

00 · What changed in v0.7.0

Eight tab commands became three

v0.6.0 was the widest this ever got — a command for every step of the loop, and a long-polling bridge to hold them together. It turned out the loop is three steps, and the agent's own scheduler already owns the waiting. Nothing below was replaced by something bigger; the work moved to whichever end already had the information.

8 tab commands → 3 tab commands · 10 bridge routes → 5 routes, none long-polling · 2 skills → 1
tabs watch The agent's own loop does the waiting — /loop in Claude Code, cron anywhere else. A blocking long-poll inside a scheduler is two schedulers.
tabs check tabs suggest already refuses a script that doesn't parse, and the extension shows the real preview before you Accept. The simulator was a third opinion nobody read.
tabs push Folded into tabs suggest. Sending a change without saying what it does was never the good half of the pair.
tabs history Deleted outright, along with the delta log it read. It was the only place the CLI kept titles and URLs of tabs you had closed.
tabs prompt The skill init installs is the prompt. Printing a second copy to paste elsewhere kept four variants of it alive in the binary.
--port · --wait · --no-wait The extension hard-codes 49227, so another port was never reachable from Chrome. And suggest no longer waits at all.

Upgrading? If you have a script that calls one of the commands above, pin v0.6.0. tabbrew init deletes the old tabbrew-auto skill for you — left in place it would keep telling your agent to run tabs watch.

02 · The loop

Three steps, whether you asked once or the agent is watching

This used to be two diagrams. Answering "tidy my tabs" and running a standing watch turned out to be the same three commands — the only difference is who decides it's time to look, and that is the scheduler's job, not the CLI's.

One pass

Read, decide, offer. Then stop — even in a loop.

  1. Setup tabbrew tabs serve & + Auto mode on The bridge in a second shell, and the Auto switch in the sidepanel's Send to Claude Code card. That switch is what keeps the tabs arriving.
  2. 1 · Read tabbrew tabs list Windows, groups, tab ids, titles, URLs — in the extension's own snapshot format. Above it: what became of the last few suggestions.
  3. 2 · Decide default: do nothing Only propose a concrete win — exact duplicates, ≥3 loose tabs on one topic, about:blank clutter. A high tab count is not a problem. Stop here if the last suggestion is still PENDING.
  4. 3 · Offer tabbrew tabs suggest plan.txt --note "…" Validates, queues, returns. The note is one plain sentence in your language, destructive part first — it's the only thing most people read.
  5. 4 · You Accept / Deny A card in the panel: the note, the preview, two buttons. Deny takes an optional reason. This is the only step where a tab actually changes.
  6. 5 · Next pass tabbrew tabs list Your answer is waiting at the top. A denial is a standing rule, not a retry signal.

The agent does not ask you to confirm a close in chat — the Accept card is the confirmation, and the --note is where the closes are named. Being asked twice for the same thing is how people stop reading either one.

Running it as a standing watch

Nothing new to learn — the same pass, on a timer.

# shell 1 — the bridge, until you Ctrl+C it
tabbrew tabs serve

# shell 2 — Claude Code
/loop 10m use the tabbrew-tabs skill: check my tabs
      and suggest something if there's an obvious win

/loop re-runs that prompt on its own schedule, so the CLI never has to block waiting for anything. Each pass is independent: read the state, decide, maybe offer, end the turn. A quiet pass should cost one line of output.

Most passes should do nothing. An agent that finds something to propose every ten minutes gets switched off within the hour — and the switch is a human who stopped trusting the notes.

What a pass reads

The header, then your last few answers, then the tabs. All three matter: the age tells it whether to trust the ids, and an unanswered suggestion tells it to stay quiet.

03 · Turning it on

Auto mode, from the human's side

The loop above is what the agent does. This is the other half — three things you set up once, and one decision you make each time a card appears.

You do three things. The agent does the rest.

1 Terminal Start the bridge tabbrew tabs serve It binds 127.0.0.1 and blocks until Ctrl+C — give it a second shell, never the one you're still working in. ✓ TabBrew bridge · ready on 127.0.0.1:49227
2 Chrome Switch Auto mode on sidepanel → Manage Tabs → Send to Claude Code Flip the Auto mode switch on that card. Keep the sidepanel open — closing it pauses the whole stream. ✓ tabs list starts showing fresh exports
3 Chat Ask the agent to watch "keep an eye on my tabs" That's what loads the tabbrew-tabs skill: how to read the snapshot, the rules for writing a note, and the rule that a denial is permanent. Add /loop 10m to put it on a timer. ✓ one short line per quiet pass

Then you browse. Until a card appears.

This is the entire interface. The note is one sentence written in whatever language you've been using, and it leads with anything that closes tabs — so you can decide without reading a diff.

TabBrew · suggestion

ปิดแท็บ YouTube ที่ค้าง 6 อัน แล้วรวม github 5 แท็บเป็นกลุ่ม Code

− 6 tabs close + group "Code" · 5 tabs
Accept Deny

Nothing in your browser has changed at this point. The script is only a proposal sitting in the panel.

Accept

Chrome runs the ops. This is the moment tabs actually move — the first and only one in the whole loop.

Deny — and say why

Nothing changes. The reason you type is written onto that suggestion in tabs.json, and the agent reads it on its next pass — where it becomes a standing rule: "อย่าปิด youtube เปิดฟังเพลงอยู่" means YouTube is off the table from now on, not just this once. That round trip is what stops a loop re-proposing what you just rejected.

Ignore it

You weren't looking. It sits there marked PENDING, and the agent's next pass sees that and stays quiet rather than stacking a second suggestion on top — the bridge holds exactly one, and a new one would silently replace it.

No double-confirming. The agent will not ask you in chat before closing anything — this card is the confirmation, and the note is where the closes are named. Being asked twice for the same thing is how people stop reading either prompt.

When it isn't working

What you seeWhat's actually wrong
nothing is listening on 127.0.0.1:49227 Step 1. The bridge isn't running, or that shell was closed.
tabs list warns that the snapshot is stale Step 2. Auto mode is off — or the sidepanel is closed, which pauses everything.
No rendered snapshot in this export It arrived from the developer-mode panel, or an extension build predating auto mode. Send it from the sidepanel instead.

How it stops

You say so

The loop ends when you tell it to, and any unclaimed suggestion goes with the bridge.

Ctrl+C the bridge

With tabs serve gone, tabs suggest reports nothing is listening and the loop has nowhere to put a proposal.

Three denials in a row

The agent stops on its own — that many refusals means its read of what you want is wrong, so it asks you directly instead of guessing a fourth time.

04 · The bridge

What actually crosses 127.0.0.1:49227

tabs serve is a small HTTP server with five routes and no auth token — the loopback bind is the entire boundary. Protocol 3 dropped every long poll: the extension polls on its own timer and the verdict is written to disk, so no request is ever held open across a human decision. Both ends update independently (extension via the Web Store, CLI via tabbrew update), so no route ever changes shape; new capability gets a new route.

1 · Tabs in

the extension is the only source of truth about your tabs

Extension POST /tabs bridge Saves tabs.json (mode 0600) and bumps the version. Carries the rendered snapshot the extension already built — which is why the CLI has no renderer of its own to drift from.

2 · Script out

one unclaimed script at a time; a new suggestion replaces it

tabs suggest POST /suggestion bridge Queues the script, the note, and the tab version it was written against. Also records it in tabs.json as undecided.
bridge GET /suggestion Extension A pop: claimed by exactly one poll, 204 when there's nothing. The extension asks every few seconds while the panel is open.

3 · Answer back

write it down rather than hold a socket open

Extension POST /decision bridge Carries accepted / denied / stale / failed plus the reason you typed, and the bridge writes it onto that suggestion in tabs.json. tabs list reads it back.
bridge GET /health Extension Cheap reachability: { ok, protocol: 3, tabsVersion, hasPending }. How the panel shows connected vs. not without claiming a queued script.

Gone in protocol 3

five routes removed; a shipped extension never called three of them

bridge GET /tabs?since=&wait= tabs watch The long-poll tabs watch parked on. Nothing in the browser ever called it.
bridge GET /history?limit= tabs history The delta log. Deleted with the log itself.
bridge GET /decision?id=&wait= tabs suggest The wait-for-the-verdict long-poll. Replaced by writing the verdict to tabs.json.
bridge GET /script · POST /script protocol 1 The pre-suggestion shapes. A current extension asks /suggestion first and only fell back to these on a 404, so it never notices.

Loopback bind, hardcoded

127.0.0.1 is not configurable. Anything already running as you on this machine can reach it — that's the accepted trade.

Host must match

127.0.0.1:port or localhost:port. This is the anti-DNS-rebinding gate: a rebound page's GETs are same-origin, so they carry no Origin at all.

Origin, when present

Must start chrome-extension://. Browsers always attach Origin to non-GET, so this blocks a drive-by POST /tabs. Invisible to curl.

0600 on disk

Your tabs are browsing history, not a revocable token — a token can be revoked, a history can't. The snapshot is written owner-only.

05 · TabBrew Script

Six verbs, one per line

The DSL an agent writes and tabs suggest validates before anyone sees it. Ids are Chrome tab ids straight from the snapshot. Blank lines and # comments are ignored.

DEL DEL <id> [<id> …] Close tabs. The only destructive verb — which is why the --note has to lead with them.
GROUP GROUP <id> … "Name"
GROUP <id> … @<gid> Into a new named group, or into an existing group by id.
UNGROUP UNGROUP <id> [<id> …] Pull tabs out of whatever group they're in.
PIN PIN <id> [<id> …] Pin to the front of the window.
UNPIN UNPIN <id> [<id> …] Unpin. Runs early, so a tab can be unpinned and regrouped in one script.
MOVE MOVE <id> <index> [@win=<wid>] Reposition, optionally into another window. The only single-id verb.
DEL→ UNPIN→ UNGROUP→ GROUP→ PIN→ MOVE

Line order does not matter. The extension collects every op and runs the six phases in this fixed order, whatever order they were written in. It matters when you write a MOVE: the index has to describe the strip after the five earlier phases have run, not the snapshot you read.

A script

# close two duplicates, file the rest
DEL 101 102
GROUP 103 104 "Code"
PIN 105
MOVE 106 0 @win=1
BOGUS 9

tabbrew tabs suggest

✗ 1 parse error:
  line 5: BOGUS 9
            → unknown verb "BOGUS"

Exit code 1, and nothing was queued — a script that doesn't parse never reaches the panel. Fix it and run the same command again. The real before/after preview is the one the extension draws on the Accept card, against the tabs as they are at that moment.

06 · The answer

Five things a suggestion can come back as

You won't see the answer in tabs suggest — it returns as soon as the bridge has the script. The verdict appears at the top of the next tabs list, which is also what stops an agent from queueing a second proposal on top of an unanswered one.

… PENDING Queued, and you haven't answered — the panel may not even be open. Wait. Do not send another suggestion; the bridge holds one at a time and a second one replaces the first.
✓ accepted You pressed Accept and Chrome applied the ops. Tabs really moved. Re-read the state before the next idea.
✗ denied You said no, optionally with a reason that comes straight back. A standing rule. Never re-send this one; fold the reason into every later decision.
↺ stale The tabs changed between the snapshot and the click, so the ids no longer line up. Not a rejection of the idea. Re-read the tabs and offer it again.
! failed You accepted, and Chrome refused. Nothing moved. They wanted this — fix the script, don't drop the idea. Reporting it as accepted would leave the agent building on a fiction.
07 · On disk & in the environment

Where the CLI keeps things

Two files, both owner-only, neither inside your project folder.

PathModeWhat it holds
~/.config/tabbrew/credentials.json 0600 Your login token. Re-asserted to 600 on every save. TABBREW_TOKEN overrides it.
~/.config/tabbrew/tabs.json 0600 The last export, plus the newest five suggestions and what you decided. Overwritten in place.

Down from three files. The old tabs-history.jsonl delta log is gone. It was the only place the CLI accumulated browsing history at rest — it remembered the titles and URLs of tabs you had closed, which nothing else here does. tabs.json only ever describes tabs that are open right now.

tabs.json, annotated

The whole contract between serve and the other two commands.

{
  "version": 42,          // bumped on every export
  "savedAt": "…",         // how tabs list judges staleness
  "count": 187,
  "tabs": [ … ], "groups": [ … ], "windows": [ … ],
  "snapshot": "# Cross-window: no\n# Windows\n…",
                          // rendered by the extension;
                          // tabs list prints this verbatim
  "suggestions": [        // newest first, capped at 5
    {
      "note": "ปิดแท็บ YouTube 6 อัน",
      "opCount": 7,
      "basedOn": 41,      // the version it was written against
      "decision": "denied", // null while unanswered
      "reason": "ยังดูอยู่"
    }
  ]
}

Why the ring is on disk

It is the agent's memory of what it already proposed, and it has to survive two things that happen constantly: a tab change (which rewrites the rest of the file) and a restart of tabs serve.

Keeping it here rather than in the agent's context means a denial still counts after a compaction, a new session, or a laptop reboot — and it's the reason no command needs to block waiting for you to look at the panel.

Five entries, overwritten in place. It's a working memory, not a record of your decisions.

EnvironmentDefaultWhy you'd set it
TABBREW_TOKEN—Authenticate in CI without an interactive login. Wins over the stored file.
TABBREW_SERVE_PORT49227Move the bridge, for tests. See the warning below — there is no --port flag.
TABBREW_TABS_PATH~/.config/tabbrew/tabs.jsonWhere tabs serve writes and tabs list reads.
TABBREW_BASE_URLwww.tabbrew.comPoint the same binary at staging or a local server.
TABBREW_NO_BROWSER—Print URLs instead of launching a browser (login, docs open).
NO_COLOR / TABBREW_DEBUG—Drop ANSI colour; print stack traces on unexpected errors.

About that port. The extension only ever talks to 49227 — it's baked into both manifests' host permissions. That's why --port was removed rather than kept: a bridge anywhere else is unreachable from Chrome, so the flag could only ever produce a bridge that looked fine from the terminal and dead from the browser.
